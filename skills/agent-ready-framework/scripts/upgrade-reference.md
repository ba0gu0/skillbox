# 老项目升级参考

> `generation-steps.md` 负责总流程；本文只维护老项目升级细节。

## Step 1: 盘点现有内容

从以下来源抽取真实信息：

| 来源 | 抽取内容 |
|------|---------|
| README.md | 项目描述、技术栈、安装命令 |
| package.json / go.mod / pyproject.toml | 依赖列表、脚本命令、框架 |
| docs/ 目录 | 已有文档结构、命名风格 |
| 代码目录结构 | 功能模块、架构 |
| 现有 AGENTS.md / CLAUDE.md | 已有红线、规则 |
| git log --oneline -10 | 最近完成的工作 |
| TODO.md / ISSUE 列表 | 待完成任务 |

## Step 2: 识别权威来源

| 情况 | 处理方式 |
|------|---------|
| 无 git 历史 | 从用户确认抽取已完成工作 |
| README 过期 | 标注「可能过期」，从代码重新抽取 |
| TODO 与代码冲突 | 以代码为准，标注冲突 |
| docs 多版本冲突 | 建立冲突清单，不仅凭修改时间定论 |

权威来源判断：

1. 代码文件：当前真实行为和模块边界
2. 配置文件：技术栈、命令、依赖、构建方式
3. 测试/CI：可验证命令和质量门槛
4. git 历史：最近完成过什么
5. README/docs：项目意图、设计说明、约定
6. 用户确认：补齐缺失或解决冲突

## 冲突处理

### README 过期检测

检查项：

- README 中的安装命令与 package.json/Makefile 是否一致？
- README 中的技术栈与 go.mod/package.json 是否一致？
- README 中的项目结构与实际目录是否一致？

处理方式：

```markdown
> README 可能过期（检测日期：{DATE}）
> 差异：<具体列出差异项>
> 已从代码重新抽取，待用户确认
```

### TODO 与代码冲突处理

检查项：

- TODO.md 中的“待完成”任务在代码中是否已实现？
- 代码中的功能在 TODO 中是否未列出？

处理方式：

| 任务 | 来源 | 冲突 |
|------|------|------|
| {TASK} | TODO.md | 代码中已实现，建议移除 |
| {TASK} | 代码分析 | TODO 中未列出，建议新增 |

### docs 多版本冲突处理

检查项：

- docs/ 目录是否有多个同名或类似文档？
- 各版本内容是否矛盾？

处理方式：建立冲突清单，写入 `{PROJECT_AUDIT_PATH}`。如无法判断，标记为
「待用户确认」，不直接固化为事实。

## Step 3: 判断 5 能力缺口

| 能力 | 有效判断 |
|------|---------|
| 项目说明 | 内容真实，无占位符 |
| 当前状态 | 阶段描述准确，已完成列表与 git 一致 |
| 下一步任务 | 任务真实可执行 |
| 验收标准 | 测试命令真实可用 |
| 红线规则 | 红线具体可执行 |

## Step 4: 询问缺失偏好

对于缺失的能力，只询问缺失能力相关偏好，不重复询问可从项目可靠抽取的信息。

## Step 5: 展示升级计划

```text
老项目升级分析
- 项目：从 README、配置或用户确认抽取
- 技术栈：从配置文件抽取
- 命名风格：{NAMING_STYLE}
- 5 能力检查：逐项列出完备、部分完备或缺失
- 将创建/更新：列出文件、来源和是否需用户确认
- 原则：保留已有命名风格，不静默覆盖，从真实内容抽取
```

## Step 6: 执行升级

对于每个缺失能力：

1. 从权威来源抽取真实内容。
2. 根据用户偏好生成文档。
3. 不静默覆盖已有文件；如需修改，先给出合并计划或 diff 草案。

禁止：

- 静默覆盖已有文件
- 生成空占位符内容
- 强制改为默认命名
- 预设任何工具链偏好

## Step 7: 最终验证

1. 列出本次实际写入的所有文件路径。
2. 将完整路径写入 `rules-reference.md` 示例中的 `FILES` 数组。
3. 确认 `FILES` 非空、无遗漏、无本次未写入文件。
4. 执行最终扫描、命令验证、链接验证和 5 能力完备检查。
