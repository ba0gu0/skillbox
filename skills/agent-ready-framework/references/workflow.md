# Agent-Ready 统一工作流

本文件定义四种运行模式的路由、渐进加载与统一执行阶段。
模式只改变读取深度、
写入边界和完成条件，不改变证据要求。

## 不变量

- 先盘点，再提问；先证明，再写入。
- 始终覆盖五个接手能力。
- 复用项目已有入口、命名和文档；文件名不是能力本身。
- 把不确定项留在证据矩阵或审计结果中，不把它们包装成项目事实。
- 只修改用户请求覆盖的文档范围；保留工作区中的用户改动和无关文件。
- 不创建依赖、包管理器产物或业务代码，除非当前任务另有明确授权。

## 模式路由

优先遵守用户明确指定且与当前事实相容的模式；
否则根据盘点结果自动选择。

| 模式 | 可观察信号 | 主要目标 | 默认写入边界 |
|------|------------|----------|----------------|
| `bootstrap` | 尚无实现资产 | 建立最小接手系统 | 最小文档，不造实现事实 |
| `onboard` | 有代码，文档不足 | 提取上下文 | 复用文档并补缺口 |
| `upgrade` | 接手文档不完整或冲突 | 合并并提升 | 先计划，再最小更新 |
| `audit` | 要求审计或只读检查 | 输出证据与建议 | 默认不写项目文件 |

若多个模式都可能成立，选择写入最少的模式。
只有模式差异会改变授权写入范围且无法
从请求判断时，才询问用户。

## 渐进加载

按以下层级加载，完成判断后停止，不要一次性读取全部模板和规范：

1. **核心层**：读取技能入口、本工作流和证据/权威规则。
2. **项目层**：读取适用指令、工作区状态、顶层结构、入口文档、manifest、
   测试与 CI 信号。
3. **模式层**：只读取所选模式解决当前缺口所需的参考资料。
4. **产物层**：只加载计划创建或更新的模板；已满足的能力不加载模板。
5. **专业层**：仅当 `AGENTS.md` 或同类项目指令在范围内时，
   加载项目指令集成规则；
   安全、架构、任务卡等参考也只在对应能力确有缺口时加载。

模式层使用以下最小切片：

- `bootstrap`：读取已存在的项目意图和约束；
  只为最小接手骨架加载产物参考，不加载升级冲突规则。
- `onboard`：先读实现、配置、测试、CI 和现有入口；
  判定缺口后只加载缺失能力的产物参考。
- `upgrade`：重点读取现有接手文档、适用指令、冲突和待合并内容；
  不重新执行新项目偏好问卷。
- `audit`：加载足够形成证据矩阵和缺口结论的资料；
  只有建议需要展示目标结构时才读取相关产物参考，并保持只读。

通过技能入口暴露的语义资源定位参考材料，
不依赖某个历史目录名或内部文件路径。

## 统一六阶段

四种模式都执行以下顺序：

### 一、盘点

- 确认请求目标、允许写入范围和适用指令链。
- 检查工作区改动，标出不得覆盖的用户内容。
- 识别项目形态、模块边界、已有文档入口及五个能力的承载位置。
- 读取能证明技术栈、命令、测试、部署和当前行为的项目资产。
- 记录冲突、不可访问内容和无法从仓库发现的信息；此时不要猜测。

### 二、证据矩阵

- 把将要写入的每个关键主张列为一行，而不是只罗列读过的文件。
- 为每行标注 `verified`、`derived` 或 `unconfirmed`，并记录具体来源。
- 区分“命令被声明”“命令可用”和“命令已成功执行”。
- 对冲突证据保留双方来源，不凭修改时间或提交信息武断裁决。

详细定义与权威边界见证据/权威参考。

### 三、能力缺口

逐项把五个能力标为：

- **完整**：内容真实、可发现、可执行且无关键冲突。
- **部分**：已有承载位置，但缺来源、入口、验收或当前性。
- **缺失**：新 Agent 无法找到该能力。
- **冲突**：多个来源给出不兼容结论，尚不能安全合并。

文件存在不等于能力完整；
必须检查新 Agent 能否不依赖聊天历史找到答案并行动。

### 四、最小文档计划

- 先修复或扩展已有文档；只有没有合适承载位置时才新建文件。
- 为每项计划记录目标能力、目标文档、操作、证据来源和验证方式。
- 删除不能补足能力的装饰性文档，不为了模板完整而增加文件。
- `audit` 只输出建议计划；其他模式也只实施请求授权范围内的部分。
- 涉及项目指令时，按项目指令集成规则委派内容生成，
  不在此处复制规范矩阵。

### 五、应用

- 按计划做最小、可审阅的修改，并保留现有命名、结构和有效内容。
- 更新旧文档前先处理冲突；不能裁决的事实保持 `unconfirmed`，
  并请求必要确认。
- 不把模板占位符、候选值、示例命令或未经证实的“已完成”
  写入最终文档。
- 生成多个入口时明确唯一事实来源，其他文件只做链接或范围补充。

### 六、验证

- 从项目入口追踪五个能力，确认链接存在、范围清楚且无需聊天历史。
- 对照证据矩阵复核所有技术栈、状态、任务、命令和红线主张。
- 扫描占位符、虚假路径、候选枚举、残留模板说明和不适用章节。
- 只运行安全且相关的真实验证；准确记录已运行、未运行及失败原因。
- 检查最终 diff 只包含计划内文件，未覆盖用户改动。
- 汇报写入文件、验证结果、未确认项和剩余风险。

## 提问门槛

仅当以下条件同时成立时才询问：

1. 选择会实质改变文档边界、事实权威、模块所有权或不可逆写入结果；
2. 现有指令、代码、配置、CI、文档和用户当前请求都无法回答；
3. 不存在安全、可逆且明显更保守的默认处理；
4. 不确认就无法继续产出真实结果。

应询问的例子：无法发现的产品目标；证据冲突且会改变当前状态；
monorepo 根规则与模块规则的所有权选择。
无需询问的例子：manifest 已确定的技术栈、lockfile 已确定的
包管理器、现有命名风格、CI 已声明的验证入口。

不要发送完整偏好问卷。
先完成所有可发现工作，再一次性提出最少的阻塞问题；非阻塞
未知项记录为 `unconfirmed` 后继续。

## 模式完成条件

- `bootstrap`：最小接手路径已建立；无法从空项目证明的实现信息未被伪造。
- `onboard`：可证事实已转成可发现入口，能力缺口已补齐或明确列出。
- `upgrade`：旧内容被保留或有依据地合并，冲突和迁移影响可审阅。
- `audit`：证据矩阵、能力缺口、最小计划和风险完整，
  且未发生未授权写入。
