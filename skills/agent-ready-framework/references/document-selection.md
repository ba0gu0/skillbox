## 选择原则

按能力缺口选择文档，不按项目规模复制固定套餐。

1. 复用现有权威文件优先于新增文件。
2. 链接到权威事实优先于复制事实。
3. 稳定规则与高频变化状态分开维护。
4. 一个文件可以承载多个能力；一个事实只保留一个权威副本。
5. 只有内容密度、更新频率或协作边界要求拆分时才新增文件。
6. 目录名和文件名遵循项目既有约定；新项目默认使用简洁语义名。

## 先映射能力

生成计划前先填写能力映射：

| 能力 | 当前承载文件 | 状态 | 处理 |
|------|--------------|------|------|
| project_identity | 真实路径或“无” | 完备/部分/缺失 | 复用/更新/新增 |
| current_state | 真实路径或“无” | 完备/部分/缺失 | 复用/更新/新增 |
| next_actions | 真实路径或“无” | 完备/部分/缺失 | 复用/更新/新增 |
| acceptance | 真实路径或“无” | 完备/部分/缺失 | 复用/更新/新增 |
| red_lines | 真实路径或“无” | 完备/部分/缺失 | 复用/更新/新增 |

“文件存在”不等于“能力完备”。
还要检查内容是否真实、当前、可从 Agent 入口发现。

## 文档角色

- **Agent 入口**
  - 适合承载：读前顺序、硬规则、精确命令、验证路由。
  - 生成信号：缺少统一入口，或现有入口无法覆盖目标 Agent。
- **README**
  - 适合承载：产品定位、安装、面向人的快速开始。
  - 生成信号：现有 README 缺少用户所需说明。
- **current state**
  - 适合承载：阶段、已验证事实、进行中、风险。
  - 生成信号：状态变化频繁，或入口文件开始承载动态内容。
- **next actions**
  - 适合承载：可执行任务、优先级、依赖、阻塞。
  - 生成信号：同时存在多个候选任务，或多人/多 Agent 领取。
- **Definition of Done**
  - 适合承载：跨任务稳定的完成条件和验证矩阵。
  - 生成信号：多类任务共享稳定验收规则。
- **project audit**
  - 适合承载：证据、冲突、缺口和迁移计划。
  - 生成信号：onboard/upgrade 存在非平凡冲突，或需长期追踪。
- **ADR**
  - 适合承载：长期有效且有替代方案的架构决策。
  - 生成信号：决策会约束未来实现，且理由不能从代码直接得出。
- **feature spec**
  - 适合承载：需求、范围、验收与设计。
  - 生成信号：功能跨模块、需求复杂或需异步评审。
- **task card**
  - 适合承载：单任务目标、边界、依赖和验收。
  - 生成信号：任务需独立领取、跨会话执行或并行协作。
- **handoff**
  - 适合承载：实际改动、验证、风险和未完成项。
  - 生成信号：长任务换执行者，且现有任务记录不足以恢复上下文。
- **security checklist**
  - 适合承载：项目特有的高风险检查。
  - 生成信号：存在认证、敏感数据、多租户、支付、上传或
    代码执行等边界。
- **glossary**
  - 适合承载：容易混淆且反复出现的领域术语。
  - 生成信号：同一术语存在歧义或影响接口/数据模型。

不要让 README、AGENTS、current state 和 task card 同时复制同一任务状态。

## 三种复杂度轮廓

轮廓只帮助控制文档上限，不决定必选文件。

### 紧凑轮廓

适用于单一工具、脚本、POC、个人维护项目。

- 优先用一个 Agent 入口承载项目说明、红线和验证路由。
- current state 可同时承载少量下一步任务。
- 验收规则很短时放在入口或 current state，不强制单独建 DoD。
- 默认不生成任务卡、spec、handoff、glossary 或独立安全清单。

### 标准轮廓

适用于持续维护的单体、API、前端、库或小型 monorepo。

- 动态状态通常从 Agent 入口拆出。
- 多个候选任务时拆出 next actions。
- 多种改动共享验证矩阵时拆出 DoD。
- ADR、spec 和任务卡按决策或任务复杂度创建，不预建空目录。

### 扩展轮廓

适用于多服务、多团队、长期平台或合规敏感系统。

- 根入口只保留全局硬规则和模块路由；子模块可有局部入口。
- 状态和任务按权威边界分层，明确根与模块的所有权。
- 架构、运行机制、ADR、spec、任务和 runbook 按真实边界拆分。
- 不因“Large”自动生成 lessons、patterns、glossary 或 handoff。

## 何时拆分

满足任一信号时考虑拆分：

- 稳定规则与每日变化状态混在同一文件。
- 同一文件由不同角色维护，更新时经常产生冲突。
- 新 Agent 无法快速定位任务、验收或红线。
- 单个章节需要独立评审、版本化或长期引用。
- monorepo 根规则与模块命令/边界明显不同。

拆分后必须从 Agent 入口添加真实相对链接，并指定新的权威来源。

## 何时不生成

- 没有真实内容时不生成空模板或 `.gitkeep`。
- 没有架构决策时不创建占位 ADR。
- 没有可领取任务时不伪造 backlog。
- 没有项目特有高风险边界时不创建通用安全清单。
- 仅一次短任务且现有任务记录足够时不生成 handoff。
- audit 模式默认在对话中报告；
  只有用户要求保存或需持续追踪时写审计文件。

## 推荐计划格式

在写入前形成简短计划：

| 操作 | 文件 | 补齐能力 | 内容来源 | 风险 |
|------|------|----------|----------|------|
| 复用/更新/新增 | 真实路径 | 能力键 | 真实来源 | 低/中/高 |

低风险且属于用户请求范围的改动可直接实施。
只有冲突来源选择、删除/迁移、公共接口、安全边界或部署行为等
会实质改变结果的事项才需要用户确认。
