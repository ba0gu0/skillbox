## 目标

在不破坏既有约定和用户改动的前提下，
让 5 个能力可发现、可信、无冲突。
升级不是把旧项目改造成默认目录结构，也不是重写全部文档。

## 升级前检查

1. 读取适用的上层、根目录和子目录 Agent 指令。
2. 执行 `git status --short`，记录已修改、未跟踪和冲突文件。
3. 盘点 README、Agent 入口、docs、manifest、lockfile、CI、runner 和部署配置。
4. 识别 monorepo 模块根与局部指令范围。
5. 从本技能目录的绝对路径运行 `scripts/project_inventory.py`，
   获取可复核的只读清单。

用户已有改动不是噪声。
若目标文件已修改，先读取差异并在其基础上合并。

## 建立现状映射

为 5 个能力记录：

- 当前权威文件和具体章节。
- 入口是否能到达该文件。
- 内容的证据等级与更新时间。
- 是否存在重复副本。
- 需要复用、更新、链接、拆分还是新增。

同一文件可以映射多个能力。
不要因为默认模板有五个文件就强行拆成五份。

## 识别冲突与陈旧内容

重点检查：

- README 命令与 package scripts、runner、CI 是否一致。
- 文档技术栈与 manifest/lockfile/源码是否一致。
- current state 的“已完成”是否有测试、发布或用户确认支持。
- TODO、issue、任务卡和代码是否互相冲突。
- 根入口与子目录入口是否越过各自作用域。
- 多个 Agent 入口是否维护了不同版本的同一规则。
- 相对链接是否按源文件目录解析后真实存在。

不要只凭修改时间选择文档，也不要只凭 git log 判定工作已经完成。

## 选择升级动作

优先级从高到低：

1. **复用**：现有文件已能承载能力，只补缺失内容。
2. **链接**：已有权威来源，仅从 Agent 入口增加路由。
3. **局部更新**：修正有明确证据的陈旧事实或命令。
4. **拆分**：稳定规则和动态状态混杂，或文件权责明显不同。
5. **新增**：没有合适承载位置且能力确实缺失。
6. **迁移/删除**：仅在用户明确同意且链接、入口和历史均已处理时执行。

默认保留文件名和目录。
混合命名可以提出统一建议，但不因美观自动迁移。

## 保护既有内容

以下改动在用户已要求升级、证据明确且不改变项目行为时可直接进行：

- 添加缺失的文档路由。
- 修正确定的断链或已失效命令。
- 删除本次生成留下的模板占位符。
- 把重复事实替换为指向权威来源的链接。

以下事项必须先确认：

- 在冲突的业务意图之间选择一个版本。
- 删除、重命名或批量移动现有文档。
- 合并多个仍被不同工具使用的 Agent 入口。
- 改变公共 API、数据模型、安全边界、部署流程或工具链。
- 覆盖无法安全合并的用户未提交内容。

项目文档不能授权违反更高优先级指令的例外。
发现冲突时保留高优先级规则并报告。

## 验证与交付

1. 把所有实际修改文件显式传给本技能绝对路径下的 `scripts/validate_docs.py`。
2. 为 5 个能力分别传入 `--require` 映射；同一文件可重复承载不同能力。
3. 核对相对链接、模板残留、占位符和路径越界。
4. 核对文档中的必跑命令来源；安全且相关时实际运行。
5. 重新检查 `git diff --check` 和目标文件差异，确认没有无关改动。
6. 报告能力映射、改动文件、验证结果、未运行验证、
   未确认事实和剩余风险。

审计文件只有在需要持续追踪冲突或用户要求保存时才写入仓库；
否则在交付信息中报告。
