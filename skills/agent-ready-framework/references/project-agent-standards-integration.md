# 与 Project Agent Standards 的集成

本文件定义两个技能的职责、数据交接和降级边界。
这里的“内容权威”指生成职责，不改变
系统、开发者、当前用户和适用项目指令的运行时优先级。

## 职责模型

`agent-ready-framework` 是总编排器，负责：

- 选择 `bootstrap`、`onboard`、`upgrade` 或 `audit` 模式；
- 盘点项目、建立证据矩阵并判断五个接手能力的缺口；
- 制定最小文档计划，协调各类文档的应用与最终验证；
- 保证五个接手能力彼此可导航。

`project-agent-standards` 是 `AGENTS.md` 及同类 Agent 指令文件的内容权威，
负责：

- 把适用指令层级、项目红线和读取顺序写成简洁入口；
- 只选择当前项目真实使用的语言、框架、工具链和工程规则；
- 生成真实的常用命令、按改动区域划分的验证矩阵和文档写回职责；
- 合并根级与子目录指令，避免无关技术栈和多份规则漂移。

`AGENTS.md` 是完整接手系统的入口和红线承载者之一，
不等于完整接手系统。状态、任务、
验收、ADR 或运行手册应由合适文档承载，再由入口路由。

## 可用时的协作流程

当 `AGENTS.md` 或同类 Agent 指令属于本次创建/更新范围，且
`project-agent-standards` 可发现并可读取时：

1. 由总编排器完成一次项目盘点和证据分级。
2. 把盘点结果按下方协议交给 `project-agent-standards`，
   不要要求它从零重复扫描。
3. 由 `project-agent-standards` 生成或合并项目指令内容；
   它可针对冲突或变化做局部复核。
4. 总编排器把返回内容接入其他文档，并检查链接、范围和事实来源。
5. 双方分别完成各自验证责任，再由总编排器汇总最终结果。

已有 `AGENTS.md` 时必须作为盘点输入进行合并，
保留仍有效的用户规则和局部作用域；
不得用通用模板整体替换。

## 盘点交接协议

通过结构化事实交接，不通过对方内部文件路径耦合。至少传递：

| 字段 | 内容 |
|------|------|
| scope | 项目根、目标模块、允许写入的指令文件 |
| instruction_chain | 当前已发现且适用的指令层级与作用域 |
| mode | 当前运行模式及写入边界 |
| project_shape | 项目类型、模块/monorepo 边界、部署形态 |
| verified_stack | 有证据定位的语言、框架、版本、包管理器 |
| command_evidence | 命令的声明、前提和本轮执行状态 |
| existing_agent_files | 现有入口、局部指令、兼容文件及有效内容 |
| capability_routes | 五个能力当前或计划中的承载位置 |
| constraints | 红线、不得覆盖内容、用户已确认决策 |
| conflicts | `derived`、`unconfirmed` 和相互冲突的主张 |
| requested_operation | 创建、合并、审计或保持不变 |

`project-agent-standards` 应复用这些事实。
只有字段缺失、盘点后目标文件发生变化，或生成
规则需要更精确证据时，才追加最小读取。

## 不可用时的最小降级

找不到或无法读取 `project-agent-standards` 时，
不阻塞整个接手系统，也不要自动安装
技能。总编排器只生成能由当前证据支持的最小项目入口：

- 项目身份和适用范围；
- 高优先级指令优先及不可信内容边界；
- 五个能力的真实读取路由；
- 经验证或用户明确确认的项目红线；
- 已声明且前提存在的常用命令，以及真实的执行状态；
- 最小验证映射和必要的文档写回职责。

不要在降级路径中复制语言规范矩阵、猜测框架最佳实践，
或生成不存在的命令。
将降级情况和未覆盖的专业规则写入结果摘要；
用户以后启用该技能时可单独精炼 `AGENTS.md`。

## 解耦规则

- `agent-ready-framework` 在没有 `project-agent-standards` 时仍能完成五能力编排。
- `project-agent-standards` 也能在不调用 `agent-ready-framework` 时独立创建或升级
  `AGENTS.md`。
- 两者不互相强制调用、不互相重新进入，不形成循环触发。
- 总编排器只在当前任务需要 Agent 指令产物时进行一次方向明确的协作。
- 语言矩阵、框架规则和 AGENTS 模板只由 `project-agent-standards` 维护；
  `agent-ready-framework` 不保存副本。
- 共享的是盘点数据和产物契约，不是内部目录、模板文件名或安装位置。
- 任一技能内部重组资源时，只要交接协议不变，另一方无需同步修改。

## 冲突与验证责任

`project-agent-standards` 负责验证 Agent 指令文件：

- 未包含不适用语言/框架规则、虚假命令或越权要求；
- 根规则和子目录规则作用域明确；
- 内容保持入口化，详细状态、任务和设计通过链接承载；
- 现有有效规则已保留，冲突已说明或标为未确认。

`agent-ready-framework` 负责系统级验证：

- 五个能力都可从入口找到，且没有两个相互竞争的事实来源；
- Agent 指令中的路径、命令、状态和其他接手文档一致；
- 本次写入符合模式和用户授权，未覆盖无关改动；
- 所有 `verified`、`derived`、`unconfirmed` 标签及执行结果如实汇报。

若两技能建议冲突，先按证据与指令优先级拆分问题。
内容组织由对应职责方裁决；事实和
授权仍由高优先级指令及证据决定，任何技能都不能自行越权。
