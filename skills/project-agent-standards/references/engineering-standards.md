# Engineering Standards

这些规则用于项目级 `AGENTS.md` 的裁剪。只写入项目真实需要的部分。

## 通用开发纪律

- 先读代码、配置、测试、CI、文档和最近提交，再动手。
- 工作区可能有用户改动；不得回滚、覆盖或格式化无关文件。
- 改动保持小而聚焦；不要顺手重构、顺手升级、顺手改风格。
- 不伪造验证结果；没跑的命令必须说没跑。
- 不接受幻觉 API、包名、配置项；第三方用法优先查官方文档或锁定版本。
- 不把失败测试改成通过；除非需求明确变更，否则测试是行为约束。
- 注释解释 why，不解释显而易见的 what。
- 错误处理必须包含上下文，方便定位失败原因。

## 依赖与工具链

- 使用项目已有包管理器和 lockfile，不混用。
- 新增依赖前确认维护状态、许可证、体积、漏洞、替代方案。
- 一次性工具优先 runner：`bunx`、`uvx`、`go run`、`cargo run`、`jbang`。
- 不在非 Node/Python 项目目录生成 `node_modules`、`package.json`、`pyproject.toml`。
- 不全局安装低频工具；必须安装时说明原因。
- 依赖升级要分清 patch/minor/major；major 升级阅读 changelog 和迁移说明。
- 生成代码要记录来源、命令和重新生成方式。

## 脚本规范

- 脚本必须可重复运行。
- 危险操作默认 `--dry-run`，或要求显式 `--force`。
- 输入参数、环境变量、输出、退出码要清晰。
- 临时目录使用系统临时目录，退出后清理。
- 不打印 secret，不把认证头、cookie、token 写入日志。
- shell 脚本使用 `set -euo pipefail`，正确引用变量。
- 长任务输出进度和失败点。
- 迁移脚本要可恢复、可重入或提供回滚说明。

## 代码组织

- 模块边界按业务能力、协议边界或部署边界划分，不按“utils”堆放。
- 公共抽象必须服务真实重复和稳定边界，不为一次性相似代码提前抽象。
- 单文件、单函数、嵌套层级保持可读；项目可在 AGENTS 中写硬上限。
- 业务逻辑不要塞进 HTTP handler、UI component、CLI parser 或数据库层。
- 配置、日志、错误模型、外部依赖 client 初始化要集中管理。

## 配置与秘密

- 配置加载要有默认值、校验、错误信息和非敏感摘要。
- `.env.example` 只放变量名和安全示例值。
- secret 来自 secret manager、环境变量或本地 ignored 文件。
- 不把 `.env`、cookie、token、私钥、证书、数据库 dump 提交到仓库。
- 日志、错误、测试快照、录屏、截图都要避免泄露 secret。

## 后端服务

- HTTP server 设置 read/write/idle/header timeout。
- HTTP client 设置 timeout，并传递取消信号。
- 后台任务、线程、goroutine、队列消费者必须有生命周期和关闭路径。
- 外部调用有重试策略、超时、退避、幂等性和错误分类。
- API 错误对外稳定，对内保留上下文。
- 结构化日志使用字段，不拼大段不可解析字符串。
- 健康检查区分 liveness/readiness；readiness 覆盖关键依赖。
- 文件上传限制大小、类型、路径、内容探测和存储位置。
- 流式接口默认流式处理，不无故缓存完整响应。

## 数据库与数据

- schema 变更必须走迁移，不直接手改生产库。
- 查询使用参数化方式；动态列名、排序、表名必须白名单。
- 事务边界清晰；不要在长事务里做外部网络调用。
- 批处理分批、可恢复、可观测。
- 删除和覆盖数据操作需要确认、备份或软删除策略。
- PII、用户内容、审计日志、财务数据要标注保留期限和访问边界。

## API 与集成

- 认证和授权分层；所有敏感操作服务端授权校验。
- 公开 API 需要稳定错误码、版本策略和兼容性说明。
- Webhook 验签、重放保护和幂等处理必须明确。
- 第三方 API 调用要有限流、熔断或降级策略。
- SSRF 风险请求限制协议、host、IP 段、DNS rebinding、重定向。

## 安全

- 默认拒绝：CORS、管理接口、上传、外部请求、文件读取都应白名单。
- session cookie 使用 `HttpOnly`、`Secure`、`SameSite` 和合理过期。
- CSRF 风险接口必须使用不可预测 token 或双提交 token；`SameSite` 作为
  额外防护，不能单独替代 token。
- 密码使用 Argon2id、bcrypt、scrypt 等专用算法。
- 加密使用标准库或成熟库，不自制协议。
- 用户输出按上下文编码：HTML、URL、JSON、SQL、shell、日志分别处理。
- 管理后台需要独立鉴权、审计日志、速率限制。
- Agent/AI 工具调用要隔离权限，敏感动作需要用户确认。

## AI 生成代码审查

- 核对 API 是否真实存在，签名是否匹配当前版本。
- 检查依赖是否真实、活跃、可信。
- 检查错误路径、边界条件、取消、超时、资源关闭。
- 检查是否引入 prompt injection、SSRF、命令注入、路径穿越、XSS、SQL 注入。
- 检查测试是否真的覆盖行为，而不是只验证 mock 调用。
- 检查是否把“示例代码”直接带入生产路径。

## 测试策略

- 项目 AGENTS 必须写最小验证命令。
- 单元测试覆盖纯逻辑、边界和错误路径。
- 集成测试覆盖数据库、网络、文件、队列、缓存等边界。
- E2E 只覆盖核心用户旅程，不替代低层测试。
- 安全或依赖变更运行对应审计工具。
- Snapshot 只能作为辅助；更新前确认行为合理。
- 时间和并发测试避免固定 sleep，使用 fake clock、context、同步点。

## 前端体验

- 第一屏应是实际可用体验，不用营销页替代工具。
- 控件使用熟悉形态：按钮、输入、菜单、tabs、toggle、slider、modal。
- 文本不溢出、不遮挡；移动端和桌面端都要验证。
- 可访问性覆盖 label、focus、键盘操作、aria、颜色对比。
- 异步状态要有 loading、empty、error、success、disabled。
- 用户可见文本使用项目 i18n；没有 i18n 时不要临时硬编码大量文案。
- 图标使用项目已有图标库。
- 复杂可视化、canvas、3D、地图要实际截图或像素验证。

## 部署

- 构建、发布、运行三阶段分离。
- 容器镜像固定基础版本，使用非 root 用户，减少包和权限。
- Compose/Kubernetes 配置包含健康检查、资源边界、日志策略。
- 生产部署有迁移顺序、回滚策略和配置校验。
- systemd 服务写清用户、工作目录、环境文件、重启策略、日志。
- serverless/edge 项目写清冷启动、超时、区域、环境变量和本地模拟方式。

## CI/CD

- CI 至少包含安装、格式/lint、测试、构建。
- 缓存 key 包含 lockfile；不缓存 secret。
- PR 自动化只给最小权限，防止恶意 PR 修改 workflow 泄密。
- IaC 变更必须有 plan/diff；生产 apply 通过受控流水线。
- 发布失败要能定位是构建、迁移、配置、健康检查还是外部依赖问题。

## 文档写回

需要更新文档的情况：

- 行为、API、配置、部署、数据库、权限、安全边界变化。
- 新增或废弃命令。
- 新增依赖或外部服务。
- 完成任务、阻塞任务、改变优先级。
- 做出长期架构决策。

不要为纯内部重命名或小修小补写过度文档。
